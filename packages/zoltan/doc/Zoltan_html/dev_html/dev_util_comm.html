Irregular Communication Library

For much of its communication, the Zoltan library uses the irregular communication package developed by Bruce Hendrickson and Steve Plimpton.  The package consists of a few simple functions which create communication maps, perform communication, and destroy communication maps upon completion.
The functions are listed below.
LB_comm_create:  computes a communication plan for sending objects to destination processors.
LB_comm_do:  uses a communication plan to send objects to destination processors.
LB_comm_destroy:  free memory associated with a communication plan.

The Zoltan library uses the irregular communication package in its migration-help tools and in some of the load-balancing algorithms. For example, in LB_Help_Migrate, LB_comm_create is used to develop a communication map for sending objects to be exported to their new destination processors. The data for the objects is packed into a communication buffer and sent to the other processors through a call to LB_comm_do. Simultaneously, each processor receives a communication buffer with data for objects it is importing. After the imported objects are unpacked, the communication plan is no longer needed; it is deallocated by a call to LB_comm_destroy. Developers should use the package whenever it is appropriate.

The order that messages are processed in the communication package is non-deterministic; it depends only on the order they are received by a processor. This non-deterministic logic provides the most efficient execution; the library reads messages from any processor and sets pointers to the storage for that processor sequentially in the receiving buffer. Thus, imported objects may be processed in different orders, depending on the order of their sending processors in the communication plan. For debugging, however, it is sometimes useful to impose a deterministic order upon the message processing. In such cases, the function call to debug_sort2_int_int in LB_comm_create should be used. It sorts the received messages in order of their processor numbers. Thus, imported objects are processed in order of their sending processors.
 
 

struct Comm_Obj *LB_comm_create(int n , int * proclist, MPI_Comm comm , int * nreturn); 

The LB_comm_create function sets up the communication plan in the irregular communication package. Its input is a count of objects to be sent to other processors, a list of the processors to which the objects should be sent, and an MPI communicator. It allocates and builds a communication plan that describes to which processors data will be sent and from which processors data will be received. It also computes the amount of data to be sent to and received from each processor. It returns the number of objects to be received by the processor and a pointer to the communication plan it created. In LB_Help_Migrate, for example, the number of objects to be exported by a processor and a list of the destination processors for the objects are passed to LB_comm_create; the resulting plan describes the communication patterns needed to export the objects and to import objects to establish a new decomposition. This plan is then used by calls to LB_comm_do to perform the actual communication.
 
Parameters:
    n The number of objects to be sent to other processors.
    proclist An array of size n of destination processor numbers for each of the objects to be sent.
    comm The MPI communicator for the irregular communication.
    nreturn Upon return, the number of objects to be received by the processor.
Returned Value:
   struct Comm_Obj * A pointer to the communication plan created by LB_comm_create.

 

void LB_comm_do(struct Comm_Obj *plan, char *sendbuf, int nsize, char *recvbuf); 

The LB_ comm_do function performs the communication described in a communication plan built by LB_comm_create. Using the plan, it takes a buffer of object data to be sent and the size (in bytes) of each object's data in that buffer and sends the data to other processors. In LB_Help_Migrate, for example, the send buffer is filled by LB_Help_Migrate through calls to the lb->Pack_Obj function for all objects to be exported. The LB_comm_do function also receives object data from other processors and stores it in a receive buffer. In LB_Help_Migrate, the data for objects to be imported is received in recvbuf and can then be unpacked through calls to the lb->Unpack_Obj function for each imported object. The receive buffer must be allocated by the code calling LB_comm_do using the number of received objects returned by LB_comm_create. All objects (both sent and received) are assumed to have the same data size.
 
Parameters:
    plan A pointer to a communication plan built by LB_comm_create.
    sendbuf A buffer filled with object data to be sent to other processors.
    nsize The size (in bytes) of the data for one object.
    recvbuf Upon return, a buffer filled with object data received from other processors.

 

void LB_comm_destroy(struct Comm_Obj **plan); 

The LB_comm_destroy function frees all memory associated with a communication plan created by LB_comm_create.
 
Parameters:
    plan A pointer to a communication plan built by LB_comm_create.  Upon return, plan is set to NULL.
